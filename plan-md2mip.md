# md2mip — Markdown/LaTeX MIP → MPS Parser & Solver CLI

## Goal

Build a **compiler** that translates MIP/LP models written in markdown/LaTeX
into standalone Python solver scripts. The LLM runs once at compile time to
parse ambiguous math notation. The output is a deterministic, inspectable
`.py` file that:

- Takes data input via YAML/JSON (abstract model pattern)
- Loads data into numpy arrays for minimal overhead
- Builds the HiGHS model using bulk `addRows`/`addCols` per constraint group
- Solves and reports results — no LLM or API key needed at runtime

## Architecture

```
                COMPILE TIME (LLM involved, one-shot)
┌─────────────────┐     ┌──────────────┐     ┌──────────────────────┐
│ Markdown + LaTeX │────▶│ Claude API   │────▶│ model_solver.py      │
│ (lossy, varied)  │     │ (parse once) │     │ (standalone, no LLM) │
└─────────────────┘     └──────────────┘     └──────────────────────┘

                RUN TIME (no LLM, runs anywhere)
┌──────────┐     ┌──────────────────────┐     ┌──────────────┐
│ data.yaml│────▶│ python model_solver.py│────▶│ solution.json│
│ data.json│     │   (numpy + highspy)   │     │ / stdout     │
└──────────┘     └──────────────────────┘     └──────────────┘
```

The generated script is the artifact — version it, inspect it, modify it by hand,
run it in CI, ship it as a microservice. The markdown source is the spec.

## Generated Code Pattern

The compiled Python script should follow this structure. Each constraint group
becomes a setup function that does bulk insertion via numpy arrays and a single
`addRows` call.

```python
#!/usr/bin/env python3
"""Transportation model — generated by md2mip from transportation.md"""

import argparse
import json
import sys

import highspy
import numpy as np
import yaml


def load_data(path: str) -> dict:
    """Load data from YAML or JSON into numpy arrays."""
    with open(path) as f:
        if path.endswith(".json"):
            raw = json.load(f)
        else:
            raw = yaml.safe_load(f)

    data = {}
    data["I"] = np.array(raw["I"])                    # set: sources
    data["J"] = np.array(raw["J"])                    # set: destinations
    data["c"] = np.array(raw["c"], dtype=np.float64)  # |I| x |J| cost matrix
    data["s"] = np.array(raw["s"], dtype=np.float64)  # |I| supply
    data["d"] = np.array(raw["d"], dtype=np.float64)  # |J| demand
    return data


def build_model(data: dict) -> highspy.Highs:
    n_i, n_j = len(data["I"]), len(data["J"])

    h = highspy.Highs()
    h.silent()

    # --- Variables: x[i,j] continuous >= 0 ---
    # Flat layout: var index = i * n_j + j
    n_vars = n_i * n_j
    costs = data["c"].flatten()  # objective coefficients
    h.addVars(n_vars, np.zeros(n_vars), np.full(n_vars, np.inf))
    h.changeColsCost(np.arange(n_vars, dtype=np.int32), costs)

    # --- Constraint group: supply (one row per i) ---
    setup_supply_constraints(h, data, n_i, n_j)

    # --- Constraint group: demand (one row per j) ---
    setup_demand_constraints(h, data, n_i, n_j)

    h.changeObjectiveSense(highspy.ObjSense.kMinimize)
    return h


def setup_supply_constraints(h, data, n_i, n_j):
    """sum_j x[i,j] <= s[i] for all i in I — one addRows call."""
    n_cons = n_i
    row_lower = np.full(n_cons, -np.inf)
    row_upper = data["s"]

    # CSR format: each row has n_j nonzeros
    starts = np.arange(0, n_cons * n_j + 1, n_j, dtype=np.int32)
    indices = np.tile(np.arange(n_j, dtype=np.int32), n_i) + \
              np.repeat(np.arange(n_i, dtype=np.int32) * n_j, n_j)
    values = np.ones(n_cons * n_j)

    h.addRows(n_cons, row_lower, row_upper, n_cons * n_j, starts, indices, values)


def setup_demand_constraints(h, data, n_i, n_j):
    """sum_i x[i,j] >= d[j] for all j in J — one addRows call."""
    n_cons = n_j
    row_lower = data["d"]
    row_upper = np.full(n_cons, np.inf)

    starts = np.arange(0, n_cons * n_i + 1, n_i, dtype=np.int32)
    indices = np.zeros(n_cons * n_i, dtype=np.int32)
    for j in range(n_j):
        for i in range(n_i):
            indices[j * n_i + i] = i * n_j + j
    values = np.ones(n_cons * n_i)

    h.addRows(n_cons, row_lower, row_upper, n_cons * n_i, starts, indices, values)


def solve_and_report(h: highspy.Highs, data: dict):
    h.run()
    status = h.getInfoValue("primal_solution_status")[1]
    obj = h.getInfoValue("objective_function_value")[1]

    n_i, n_j = len(data["I"]), len(data["J"])
    sol = np.array(h.getSolution().col_value).reshape(n_i, n_j)

    print(f"Status: {'optimal' if status == 2 else status}")
    print(f"Objective: {obj:.4f}")
    print(f"Solution:")
    for i in range(n_i):
        for j in range(n_j):
            if sol[i, j] > 1e-6:
                print(f"  x[{data['I'][i]},{data['J'][j]}] = {sol[i,j]:.2f}")


if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("data", help="Path to data file (YAML or JSON)")
    args = parser.parse_args()

    data = load_data(args.data)
    h = build_model(data)
    solve_and_report(h, data)
```

### Key Design Principles for Generated Code

1. **One setup function per constraint group** — named after the constraint
   (e.g., `setup_supply_constraints`). Each calls `addRows` exactly once.
2. **Flat variable indexing** — multi-dimensional variables use a flat layout
   with a clear mapping comment (e.g., `var index = i * n_j + j`).
3. **CSR format for bulk insertion** — `addRows(n, lower, upper, nnz, starts, indices, values)`
   is the fast path in HiGHS. Build the sparse structure with numpy.
4. **numpy everywhere** — data loads into numpy arrays immediately. No Python
   loops over data during model construction where avoidable.
5. **Self-contained** — the generated script has zero dependency on md2mip at
   runtime. Only `highspy`, `numpy`, `pyyaml` (and `json` stdlib).
6. **Inspectable** — a human OR practitioner should be able to read the generated
   code and verify it matches the formulation.

### Data File Format

For the transportation example above, the corresponding `data.yaml`:

```yaml
I: ["factory1", "factory2"]
J: ["warehouse1", "warehouse2", "warehouse3"]
c:
  - [4, 8, 1]
  - [7, 2, 3]
s: [30, 50]
d: [20, 25, 35]
```

Or equivalently `data.json`:

```json
{
  "I": ["factory1", "factory2"],
  "J": ["warehouse1", "warehouse2", "warehouse3"],
  "c": [[4, 8, 1], [7, 2, 3]],
  "s": [30, 50],
  "d": [20, 25, 35]
}
```

## Model IR Schema (Intermediate)

The LLM produces this JSON as an intermediate step during compilation. The code
generator then translates IR → Python. This is internal to md2mip and not
exposed to the user, but useful for debugging.

```json
{
  "name": "transportation",
  "sets": {
    "I": { "description": "sources", "elements": ["factory1", "factory2"] },
    "J": { "description": "destinations", "elements": ["warehouse1", "warehouse2", "warehouse3"] }
  },
  "parameters": {
    "c": { "indices": ["I", "J"], "description": "unit transport cost", "data": [[4,8,1],[7,2,3]] },
    "s": { "indices": ["I"], "description": "supply capacity", "data": [30, 50] },
    "d": { "indices": ["J"], "description": "demand", "data": [20, 25, 35] }
  },
  "variables": {
    "x": {
      "indices": ["I", "J"],
      "type": "continuous",
      "lower_bound": 0,
      "upper_bound": null,
      "description": "amount shipped"
    }
  },
  "objective": {
    "sense": "minimize",
    "expression": "sum(c[i,j] * x[i,j] for i in I for j in J)"
  },
  "constraints": {
    "supply": {
      "for_all": ["i in I"],
      "expression": "sum(x[i,j] for j in J) <= s[i]"
    },
    "demand": {
      "for_all": ["j in J"],
      "expression": "sum(x[i,j] for i in I) >= d[j]"
    }
  }
}
```

## Implementation Plan

### Phase 1: Core Compiler

1. **`md2mip compile <model.md> -o solver.py`** — the main command. Sends markdown
   to Claude API, gets IR, generates a standalone Python solver script.
2. **`md2mip compile <model.md> --ir-only`** — emit just the IR JSON (for debugging).
3. **`md2mip run <model.md> --data data.yaml`** — compile + immediately run (convenience).

### Phase 2: Extensions

- `md2mip ocr <image.png> -o model.md` — OCR a photo of a model (Claude vision) → markdown
- `md2mip compile --target mps` — generate MPS file instead of Python script
- Multiple data scenarios / batch solve in generated script
- Solution report generation (format back into markdown tables)

### Dependencies

md2mip itself (compile time):
```
pip install anthropic click
```

Generated scripts (run time — no md2mip needed):
```
pip install highspy numpy pyyaml
```

## Test Models

Below are 6 textbook models written in deliberately varied markdown/LaTeX
notation. A robust parser should handle all of these. They range from clean
academic notation to sloppy/informal styles.

---

### Model 1: Transportation Problem (Clean Academic Style)

A standard transportation problem with explicit set and parameter definitions.

#### Sets

- $I = \{1, 2\}$ — Sources (factories)
- $J = \{1, 2, 3\}$ — Destinations (warehouses)

#### Parameters

| | $j=1$ | $j=2$ | $j=3$ |
|---|---|---|---|
| $c_{1j}$ | 4 | 8 | 1 |
| $c_{2j}$ | 7 | 2 | 3 |

- Supply: $s_1 = 30, \; s_2 = 50$
- Demand: $d_1 = 20, \; d_2 = 25, \; d_3 = 35$

#### Variables

$x_{ij} \geq 0 \quad \forall i \in I, \; j \in J$

#### Formulation

$$\min \sum_{i \in I} \sum_{j \in J} c_{ij} x_{ij}$$

subject to:

$$\sum_{j \in J} x_{ij} \leq s_i \quad \forall i \in I$$

$$\sum_{i \in I} x_{ij} \geq d_j \quad \forall j \in J$$

#### Expected Solution

Optimal objective value: **110**

---

### Model 2: 0-1 Knapsack (Terse / Shorthand Style)

Sloppy notation — no explicit set definitions, inline data, mixed conventions.

We have 5 items. Knapsack capacity W=15.

Values: $v = (4, 2, 10, 1, 2)$

Weights: $w = (12, 1, 4, 1, 2)$

Pick items to maximize value:

$$\max \sum_{i=1}^{5} v_i x_i$$
$$\text{s.t.} \quad \sum_{i=1}^{5} w_i x_i \leq W$$
$$x_i \in \{0, 1\}$$

#### Expected Solution

Select items 2, 3, 4, 5 → value = **15**, weight = 8

---

### Model 3: Facility Location (Verbose / Textbook Prose Style)

**Uncapacitated Facility Location Problem (UFLP)**

Consider a set of potential facility locations $I = \{A, B, C\}$ and a set of
customers $J = \{1, 2, 3, 4\}$. Opening facility $i$ incurs a fixed cost $f_i$.
Serving customer $j$ from facility $i$ costs $c_{ij}$ per unit. Each customer
has a demand of 1 unit that must be satisfied by exactly one facility.

**Fixed costs:**
- $f_A = 100, \quad f_B = 150, \quad f_C = 120$

**Service costs $c_{ij}$:**

|  | 1 | 2 | 3 | 4 |
|--|---|---|---|---|
| A | 10 | 25 | 40 | 15 |
| B | 30 | 5 | 10 | 20 |
| C | 20 | 15 | 20 | 10 |

**Decision variables:**
- $y_i = 1$ if facility $i$ is opened, 0 otherwise
- $x_{ij} = 1$ if customer $j$ is served from facility $i$, 0 otherwise

**Minimize** total cost:

$$\min \; \sum_{i \in I} f_i y_i + \sum_{i \in I} \sum_{j \in J} c_{ij} x_{ij}$$

**Subject to:**

Each customer is served by exactly one facility:
$$\sum_{i \in I} x_{ij} = 1 \quad \forall j \in J$$

A customer can only be served from an open facility:
$$x_{ij} \leq y_i \quad \forall i \in I, \; j \in J$$

$x_{ij} \in \{0, 1\}, \; y_i \in \{0, 1\}$

#### Expected Solution

Open facilities A and C. Serve: 1←A, 2←C, 3←C, 4←C. Total cost = 100 + 120 + 10 + 15 + 20 + 10 = **275**

### Model 4: Production Planning / Lot Sizing (Time-Indexed, Dense Style)

Multi-period production planning. This uses a common style where the time set is
written as a range and parameters are given as vectors.

**Sets:** $T = \{0, 1, \ldots, 7\}$ (8 weeks), but production starts at $t=0$.

**Parameters:**
- Demand: $d = [10, 15, 20, 18, 12, 25, 22, 16]$ (indexed 0..7)
- Production cost: $p_t = 5 \;\; \forall t$
- Holding cost: $h = 2$ per unit per period
- Setup cost: $K = 50$
- Max production: $M = 40$
- Initial inventory: $I_0^{\text{init}} = 5$

**Variables:**
- $x_t \geq 0$ — units produced in period $t$
- $I_t \geq 0$ — inventory at end of period $t$
- $z_t \in \{0,1\}$ — whether production occurs in $t$

$$\min \sum_{t \in T} \left( p_t x_t + h \cdot I_t + K \cdot z_t \right)$$

s.t.

$$I_t = I_{t-1} + x_t - d_t \quad \forall t \in T$$

where $I_{-1} = I_0^{\text{init}} = 5$.

$$x_t \leq M \cdot z_t \quad \forall t \in T$$

$$x_t \geq 0, \quad I_t \geq 0, \quad z_t \in \{0,1\} \quad \forall t \in T$$

#### Expected Solution

Should find a plan that batches production to avoid setup costs where possible.
Feasibility check: total demand = 138, initial inventory = 5, so need to produce ≥ 133.

### Model 5: Assignment Problem (Matrix-Heavy, Minimal LaTeX)

Assign 4 workers to 4 tasks. Cost matrix:

```
     T1  T2  T3  T4
W1 [  9   2  7   8 ]
W2 [  6   4  3   7 ]
W3 [  5   8  1   8 ]
W4 [  7   6  9   4 ]
```

min sum_i sum_j c_ij * x_ij

Each worker does exactly one task:
  sum_j x_ij = 1  for all i

Each task gets exactly one worker:
  sum_i x_ij = 1  for all j

x_ij in {0,1}

#### Expected Solution

W1→T2, W2→T3, W3→T1 (or T3?), W4→T4 — optimal cost = **13**

Assignments: W1→T2(2), W2→T3(3), W3→T1(5), W4→T4(4) = 14? Let me not
precommit, just check feasibility. The Hungarian algorithm gives the answer.

### Model 6: Blending / Diet Problem (Natural Language Heavy)

A feed company wants to make a mixture using two ingredients (Corn and Oats).
The mixture must contain at least 20% protein and at most 5% fiber.

Corn costs $0.30/kg and contains 9% protein and 2% fiber.
Oats cost $0.50/kg and contains 15% protein and 8% fiber.

They need at least 100 kg of feed.

Minimize cost. How much of each ingredient?

(This has no LaTeX at all — the parser must extract the model from prose.)

#### Expected Solution

This is a simple 2-variable LP. The protein constraint is binding.
Approximately 83.3 kg corn and 16.7 kg oats, cost ≈ $33.33.

## Evaluation Criteria

For each test model, verify:

1. **Parsing correctness** — Does the IR capture all sets, parameters, variables, constraints?
2. **Code quality** — Is the generated Python readable, correct, and using bulk operations?
3. **Data binding** — Does the generated script correctly load YAML/JSON and map to numpy arrays?
4. **Bulk insertion** — Does each constraint group use a single `addRows` call with CSR format?
5. **Solution quality** — Does the generated script solve to the expected optimal?
6. **Standalone** — Does the generated script run without md2mip installed?

## Stretch Goals / Things to Explore

- **Round-trip**: generated script formats solution back into markdown tables
- **Model diff**: compare two markdown model formulations structurally via IR
- **Parametric analysis**: generate a script with `--sweep` support built in
- **Infeasibility diagnosis**: generate IIS extraction code in the script
- **Abstract model library**: save IRs and re-compile with different code targets
- **Flowty integration**: detect network structure → generate Flowty-based solver instead of HiGHS
- **MPS export target**: `md2mip compile --target mps` for solver-agnostic output
- **Validation mode**: compile, run with inline data, compare to expected solution in markdown

## Notes on the LLM Parsing Prompt

The system prompt for the Claude API call that does markdown → IR should:

1. Explain the IR schema with a complete worked example
2. Emphasize that ALL data must be extracted (don't summarize or skip values)
3. Handle ambiguity explicitly:
   - Distinguish parameters vs. variables (if it has data, it's a parameter)
   - Infer index sets from context when not declared
   - Treat $\{0,1\}$ domain as binary, $\geq 0$ as continuous
4. Ask the LLM to flag anything it's uncertain about in a `"warnings"` field
5. Handle edge cases:
   - Parameters given as tables vs. inline vs. vectors
   - Constraints written with $\forall$ vs. "for all" vs. implied
   - Objectives with $\min$ vs. $\text{minimize}$ vs. prose
   - Summation with $\sum_{i=1}^{n}$ vs. $\sum_{i \in I}$
   - Time-indexed models with $t-1$ references (inventory balance)

**Two-stage compilation option:** For complex models, consider a two-LLM-call
approach: (1) markdown → IR JSON, (2) IR JSON → Python code. The second call
gets the IR + the target code template and generates the setup functions. This
lets you validate the IR before code generation.
